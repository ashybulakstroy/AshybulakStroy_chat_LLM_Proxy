# P2P Subproject Specification

## Назначение документа

Этот документ фиксирует мое текущее понимание будущего `P2P`-подпроекта внутри `AshybulakStroy AI HUB`.

Это не описание текущей реализации.
Это целевая архитектурная спецификация для следующего слоя развития проекта.

Документ нужен, чтобы:

- отделить `P2P`-логику от текущего `provider proxy`
- зафиксировать роли `MASTER`, `NODE`, `PEER`, `MASTER_CACHE`
- определить границы будущего файла `app/p2p_service.py`
- избежать смешивания `peer routing` с уже существующим `provider routing`

## Контекст текущего проекта

Сейчас проект `Prj_8_LLM_Proxy` уже умеет:

- принимать OpenAI-compatible запросы
- маршрутизировать их между upstream-провайдерами
- собирать live/fallback rate limits
- хранить runtime-статус диспетчера
- переключать режимы `FAST` и `LOAD_BALANCE`
- показывать состояние через admin endpoints

Текущая система уже частично похожа на `MASTER`, но единицей маршрутизации сейчас является не `peer`, а `provider`.

То есть сейчас диспетчер выбирает:

- `groq`
- `openrouter`
- `gemini`
- `cerebras`
- и другие upstream-провайдеры

Но пока не выбирает между:

- `Node-МойПК`
- `Node-Мама`
- `Node-Офис`
- `Node-Сервер`

Из-за этого текущий проект является централизованным proxy/router, а не полноценной `P2P`-сетью.

## Главная идея P2P-подпроекта

`P2P`-подпроект должен превратить текущий AI HUB из "роутера провайдеров" в "оркестратор распределенной сети узлов".

Итоговая цепочка должна выглядеть так:

`Client -> Master -> Peer -> Local Provider Router -> Upstream Provider`

Важная мысль:

- `provider routing` отвечает за выбор внешнего LLM-провайдера внутри одного узла
- `peer routing` отвечает за выбор подходящего узла внутри сети

Эти два уровня не должны смешиваться в одной логике.

## Базовые сущности

### NODE

`Node` — это установленный экземпляр AI HUB на конкретной машине.

Примеры:

- домашний ПК
- мамин ноутбук
- рабочий компьютер
- VPS
- сервер в Docker

Каждый `Node` может иметь:

- свой набор API-ключей
- свой набор доступных провайдеров
- свои локальные модели
- свои лимиты RPM/TPM
- свои сетевые характеристики
- свою политику участия в сети

`Node` — это программная и инфраструктурная оболочка.

### PEER

`Peer` — это не отдельный тип машины, а статус `Node`, когда он:

- зарегистрирован в сети
- доступен по health-check
- готов принимать внешние задачи
- разрешил делиться частью своих ресурсов

То есть:

- каждый `Peer` является `Node`
- но не каждый `Node` в каждый момент времени является `Peer`

Причины, по которым `Node` может временно перестать быть `Peer`:

- нет интернета
- бан/ошибка провайдера
- закончились лимиты
- ручная пауза
- высокая локальная нагрузка
- неуспешный heartbeat

### MASTER

`Master` — это центральный координатор сети.

Он не обязан сам иметь API-ключи.
Его главная роль — принимать решение, куда пойдет запрос.

`Master` отвечает за:

- discovery новых узлов
- handshake и регистрацию
- хранение live-карты сети
- health tracking
- fairness
- peer selection
- execution strategy
- ведение истории маршрутизации
- интеграцию с `MASTER_CACHE`

### MASTER_CACHE

`MASTER_CACHE` — это централизованная память сети на стороне `Master`.

Peers не должны знать о нем и не должны управлять им напрямую.

Причины:

- меньше связности
- проще поведение peer-узлов
- легче контроль консистентности
- дешевле повторные запросы

Роль `MASTER_CACHE`:

- проверка повторяемых запросов
- экономия лимитов peer-узлов
- ускорение типовых ответов
- сохранение статистики маршрутизации

## Архитектурный принцип

### Что остается локальным provider-level слоем

Эти части уже есть в проекте и должны остаться локальными для конкретного узла:

- `app/router_service.py`
- `app/providers/openai_provider.py`
- `app/rate_limits.py`

Их задача:

- знать локальные провайдеры
- знать локальные ключи
- вызывать upstream API
- отслеживать лимиты локальной машины

### Что добавляется как новый peer-level слой

Новый `P2P`-подпроект должен жить отдельно и отвечать за:

- discovery
- peer registry
- handshake
- heartbeat
- peer health
- peer reputation
- распределение задач между узлами
- fair sharing
- remote execution

То есть:

- `router_service.py` выбирает `provider`
- `p2p_service.py` выбирает `peer`

## Граница нового подпроекта

Первый логический контейнер для новой логики:

- `app/p2p_service.py`

Почему это правильно:

- `routes.py` уже перегружен endpoint-логикой
- `router_service.py` уже отвечает за provider dispatch
- `admin_ui.py` уже отвечает за UI
- `rate_limits.py` уже отвечает за лимиты провайдеров

Если смешать `P2P` прямо в эти файлы, система быстро потеряет читаемость.

Поэтому `app/p2p_service.py` должен стать самостоятельным orchestration-слоем.

Позже его можно разрезать на:

- `app/p2p/registry.py`
- `app/p2p/discovery.py`
- `app/p2p/health.py`
- `app/p2p/dispatch.py`
- `app/p2p/cache.py`
- `app/p2p/transport.py`

Но стартовать лучше с одного файла и четкой внутренней структурой.

## Роли будущих слоев

### 1. HTTP API Layer

Это существующие `routes.py` и FastAPI endpoints.

Роль:

- принять клиентский запрос
- валидировать payload
- передать задачу в orchestration layer
- вернуть результат в OpenAI-compatible формате

### 2. P2P Orchestration Layer

Это будущий `app/p2p_service.py`.

Роль:

- понять, нужен ли локальный запуск или remote dispatch
- выбрать peer
- применить fairness и policy
- проверить cache
- оформить трассировку выбора

### 3. Local Provider Routing Layer

Это существующий `ProviderRouter`.

Роль:

- выбрать локального upstream-провайдера
- выполнить запрос
- вернуть ответ

### 4. Upstream Provider Layer

Это `OpenAIProvider` и будущие provider adapters.

Роль:

- отправить запрос в OpenAI-compatible API
- получить ответ
- записать headers и ошибки

## Основная идея маршрутизации

Будущий AI HUB должен иметь два независимых решения:

### Решение 1. На каком узле выполнять задачу

Это делает `P2P`-слой.

Он отвечает на вопрос:

"Кто из peer-узлов должен обработать задачу?"

### Решение 2. Через какого провайдера внутри узла выполнять задачу

Это делает локальный `ProviderRouter`.

Он отвечает на вопрос:

"Какой upstream-провайдер на выбранном узле лучше подойдет?"

Именно эта двухступенчатая модель является фундаментом будущей сети.

## Будущие роли объектов внутри P2P

Ниже не код, а целевая модель сущностей.

### NodeRecord

Описывает установленный узел в сети.

Должен содержать:

- `node_id`
- `label`
- `base_url`
- `host`
- `port`
- `role`
- `registered_at`
- `last_seen_at`
- `status`
- `local_network_id`
- `public_network_id`
- `software_version`
- `capabilities`
- `routing_policy`

### PeerRecord

Описывает текущую peer-сессию узла.

Должен содержать:

- `peer_id`
- `node_id`
- `is_accepting_tasks`
- `shared_rpm_ratio`
- `shared_tpm_ratio`
- `active_sessions`
- `health_score`
- `reputation_score`
- `last_heartbeat_at`
- `last_error`
- `cooldown_until`

### PeerCapability

Описывает, что реально умеет конкретный peer.

Должен содержать:

- `providers`
- `models`
- `task_types`
- `supports_chat`
- `supports_embeddings`
- `supports_audio`
- `supports_image`
- `supports_video`
- `supports_local_models`

### TaskDescriptor

Описывает нормализованную задачу для dispatch.

Должен содержать:

- `task_id`
- `task_type`
- `requested_provider`
- `requested_model`
- `requested_mode`
- `estimated_tokens`
- `priority`
- `metadata`
- `cache_key`

### DispatchDecision

Описывает итоговое решение диспетчера.

Должен содержать:

- `decision_id`
- `selected_peer`
- `selected_provider`
- `selected_model`
- `selection_policy`
- `decision_reason`
- `used_cache`
- `latency_class`
- `fallback_chain`

## Discovery

### Цель discovery

Сеть должна узнавать о новых `Node` и поддерживать карту живых `Peer`.

### Что должен обеспечивать discovery

- обнаружение новых узлов
- повторную регистрацию после рестарта
- обновление адресов и capability snapshot
- удаление мертвых peer-узлов из активной ротации

### Возможные режимы discovery

#### Local Discovery

Для одной локальной сети:

- mDNS
- Bonjour
- UDP broadcast
- LAN announce

#### Coordinated Discovery

Для нескольких сетей:

- signal server
- static peer list
- manual bootstrap URL

### Моя интерпретация

Для этого проекта разумно заложить абстракцию discovery, но на первом этапе не зашивать mDNS глубоко в core.

Сначала достаточно предусмотреть логический протокол:

- `peer announce`
- `master accept`
- `peer sync`
- `heartbeat`

А транспорт discovery можно добавлять позже.

## Handshake

Handshake нужен не для красоты, а для доверия и синхронизации состояния.

На handshake `Peer` должен сообщать:

- свою identity
- URL для обратной связи
- доступные task types
- локальные провайдеры
- локальные модели
- лимиты
- долю ресурсов, которую он готов отдавать сети

`Master` в ответ должен:

- подтвердить регистрацию
- выдать внутренний peer identifier
- зафиксировать peer policy
- добавить peer в network map

## Heartbeat

Heartbeat нужен для поддержания статуса peer-узла.

Heartbeat должен сообщать:

- что peer жив
- готов ли он принимать задачи
- сколько активных задач уже идет
- каковы остатки лимитов
- есть ли свежие ошибки
- каков локальный health snapshot

Если heartbeat не приходит в срок:

- peer должен быть переведен в `degraded` или `offline`
- peer исключается из ротации

## Health model

`Health Score` — это один из центральных механизмов сети.

Он не должен зависеть от одного параметра.

Я понимаю его как композицию:

- доступность heartbeat
- успешность последних запросов
- свежесть ошибок
- latency
- остаток лимитов
- локальная нагрузка
- ручной статус peer-а

### Что должно понижать health score

- таймауты
- частые 429
- частые 5xx
- постоянные network errors
- отсутствие heartbeat
- нулевые shared limits
- слишком много активных сессий

### Что должно повышать или поддерживать health score

- стабильные успешные ответы
- низкая latency
- достаточные остатки лимитов
- малое число активных задач
- стабильный heartbeat

## Fairness

Это ключевая часть вашей идеи.

Каждый peer должен иметь право ограничить, сколько своей мощности он отдает сети.

### Что означает fairness в этой сети

- peer не обязан отдавать 100% своих лимитов
- peer может участвовать частично
- master обязан уважать этот лимит

### Примеры

- домашний peer делится 80% лимита
- мамин peer делится только 20%
- офисный peer делится только ночью
- серверный peer делится всегда

### Следствие

`Master` должен оперировать не полными лимитами узла, а только `shared capacity`.

То есть в маршрутизации должны участвовать:

- не просто `remaining_rpm`
- а `shared_remaining_rpm`

## Репутация peer-а

Помимо health score, в системе нужен более инерционный показатель — `reputation_score`.

Разница:

- `health_score` отражает текущее состояние
- `reputation_score` отражает историческую надежность

Это защитит сеть от peer-узлов, которые:

- часто "оживают и падают"
- часто дают плохие ответы
- часто создают ложную готовность

## Режимы диспетчеризации

Текущий проект уже знает:

- `FAST`
- `LOAD_BALANCE`

В P2P-системе я понимаю их так.

### FAST

Задача уходит на peer с минимальной ожидаемой задержкой.

Подходит для:

- чата
- интерактивных сценариев
- быстрых классификаций

Приоритеты:

- локальная сеть
- низкая latency
- живой peer
- достаточный лимит

### LOAD_BALANCE

Задача уходит туда, где сеть меньше всего перегружена.

Подходит для:

- пакетной обработки
- массовых заданий
- фоновых задач

Приоритеты:

- низкое число активных сессий
- высокий shared remaining rpm/tpm
- равномерное распределение по peer-узлам

### LOCAL_FIRST

Это режим, который естественно напрашивается в P2P.

Сначала:

- пытаемся обработать локально

Если локальный узел не подходит:

- отправляем на внешний peer

### COST_EFFECTIVE

Еще один логичный режим для будущего.

Сначала тратим:

- бесплатные провайдеры
- дешевые модели
- peer-узлы с лучшей бесплатной емкостью

Потом:

- платные
- дорогие
- лимитно-чувствительные

## PPR: Priority Peer Routing

Ваше описание `PPR` я понимаю как приоритетную маршрутизацию по близости и качеству.

Это не просто "первый свободный peer".

Это выбор peer-а по нескольким уровням:

1. peer в той же локальной сети
2. peer с лучшей latency
3. peer с достаточным shared limit
4. peer с хорошим health score
5. peer с низкой текущей загрузкой

### Практический смысл

- меньше задержка
- меньше риск сетевого сбоя
- меньше лишнего внешнего трафика
- выше предсказуемость

## Execution flow

Ниже моя интерпретация желаемого жизненного цикла запроса.

### Шаг 1. Прием запроса

Клиент отправляет стандартный OpenAI-compatible запрос в `Master`.

### Шаг 2. Нормализация

`Master` превращает запрос в `TaskDescriptor`.

### Шаг 3. Cache lookup

`Master` проверяет `MASTER_CACHE`.

Если hit:

- ответ отдается сразу

Если miss:

- запускается peer selection

### Шаг 4. Peer selection

`Master` выбирает peer по текущему mode и policy.

### Шаг 5. Remote or local execution

Если выбран текущий узел:

- вызывается локальный `ProviderRouter`

Если выбран внешний peer:

- запрос уходит на peer transport API

### Шаг 6. Local provider selection inside peer

Выбранный peer локально решает:

- каким провайдером
- какой моделью
- с какими локальными fallback-ами

обработать задачу.

### Шаг 7. Ответ

Peer возвращает результат `Master`-узлу.

### Шаг 8. Cache save

`Master` сохраняет ответ в `MASTER_CACHE`.

### Шаг 9. Return to client

Клиент получает ответ и в идеале не знает, где именно он был выполнен, кроме случаев, когда это явно показывается в debug/admin.

## Peer transport

Внутренний транспорт между `Master` и `Peer` должен быть отдельной логической зоной.

Его задача:

- передавать задачу на peer
- передавать результат обратно
- фиксировать метаданные маршрута

Важно:

`Peer transport` не равен upstream provider transport.

Это другой слой.

Я бы закладывал для него отдельные понятия:

- internal request envelope
- internal response envelope
- peer auth
- timeout policy
- retry policy

## Безопасность

В P2P-системе доверие между узлами нельзя оставлять неформальным.

Минимально должны быть предусмотрены:

- peer identity
- whitelist/registration control
- shared secret или token
- ограничение, какие peers могут принимать какие задачи
- защита от ложного heartbeat
- защита от неавторизованной регистрации

Важно:

peer может быть "дружественным", но все равно должен проходить формальный handshake и validation.

## Кодировки и данные

С учетом смешанных ОС и прошлых проблем с текстом, сеть должна считать обязательными правилами:

- весь внутренний обмен только в `UTF-8`
- только валидный JSON
- никакого "сырого" текстового протокола
- четкие поля статусов и ошибок

Это особенно важно для:

- русского текста
- казахского текста
- mixed prompt payloads
- логов и диагностики

## Admin и observability

Существующая админка уже дает хорошую базу.

В будущем `P2P` должен добавить в dashboard:

- список peer-узлов
- статус каждого peer-а
- последний heartbeat
- health score
- reputation score
- active sessions by peer
- shared limits by peer
- route history
- cache hit/miss
- причины выбора peer-а

То есть admin должен показывать уже не только:

- `provider state`

но и:

- `network state`

## Что должно остаться вне P2P core

В `p2p_service.py` не должно попадать все подряд.

Я не ожидаю там:

- FastAPI endpoint-ы
- HTML/CSS/JS админки
- низкоуровневый provider HTTP client
- raw file UI cache logic

Это должны оставаться отдельные слои.

## Что должно войти в app/p2p_service.py

На первом этапе я ожидаю там только orchestration-сущности.

Пример логических зон файла:

- network registry
- peer state management
- heartbeat processing
- health scoring
- fairness policy
- peer selection
- dispatch decisions
- local vs remote execution entrypoints
- cache facade hooks

## Как новый слой должен стыковаться с текущими файлами

### routes.py

Должен:

- принимать API запрос
- вызывать `p2p_service`

Не должен:

- сам решать, какой peer выбрать

### router_service.py

Должен:

- остаться локальным маршрутизатором провайдеров

Не должен:

- знать про discovery peers
- знать про heartbeat peers
- знать про fairness между узлами

### rate_limits.py

Должен:

- продолжать хранить локальные лимиты провайдеров

Позже может:

- отдавать агрегированный snapshot для peer health

### admin_ui.py и admin cache

Должны:

- читать состояние P2P-слоя
- визуализировать network map

Но не должны:

- содержать бизнес-логику диспетчеризации

## Главная архитектурная развилка

Самая важная вещь, которую я сейчас понимаю так:

нельзя строить `P2P`, просто расширяя текущий список провайдеров.

`Peer` — это не provider.

Потому что:

- у одного peer может быть много providers
- peer имеет собственный health
- peer имеет собственную fairness policy
- peer имеет сетевую близость
- peer может быть локальным или удаленным
- peer может временно выйти из ротации независимо от провайдера

Поэтому `P2P` должен быть отдельной координатой системы.

## Ограничения и честные замечания

На момент написания этого документа проект еще не является реальной P2P-сетью.

В нем уже есть:

- dispatcher mindset
- rate-limit awareness
- proxy modes
- admin cache

Но в нем еще нет:

- peer registry
- peer heartbeat
- peer auth
- peer transport
- удаленного выполнения между нодами
- реального master-cache как сетевой подсистемы

Поэтому этот документ описывает целевую модель, а не утверждает, что она уже реализована.

## Итоговое понимание

Я понимаю ваш `P2P`-подпроект так:

`AshybulakStroy AI HUB` должен стать распределенной системой, где:

- `Master` управляет сетью
- `Node` является установленным экземпляром HUB
- `Peer` является живым и допущенным к работе статусом узла
- `MASTER_CACHE` экономит лимиты и ускоряет ответы
- `ProviderRouter` остается локальным слоем внутри каждого узла
- `P2P service` становится верхним orchestration-слоем над локальным provider routing

Коротко:

текущий проект уже готов быть фундаментом `Master`,
но для настоящего `P2P` ему нужен отдельный peer orchestration layer, который нельзя смешивать с текущим provider router.
